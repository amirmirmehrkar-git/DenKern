# مدل کاری تیم و نقش لید

Source: Amir provided a leadership/operating-model note from an earlier chat.

## نقش واقعی Amir

Amir نباید فقط task بدهد. نقش اصلی باید این باشد:

- Product and direction owner.

یعنی:

- چی بسازیم.
- چرا بسازیم.
- چه چیزی مهم است.

## مدل اشتباه

- "من می‌گم شما انجام بدید"

نتیجه:

- bottleneck روی Amir.
- disengagement برای تیم.
- اجرا کند.

## مدل درست

- "Amir direction می‌دهد و outcome تعریف می‌کند. بقیه ownership اجرا دارند."

اصل:

```text
task نده -> outcome بده
```

## ساختار پیشنهادی نقش‌ها

Amir (Lead / Product):

- تعریف use case.
- تعریف MVP.
- تصمیم نهایی.
- alignment.

James (Execution - Technical):

- ساخت logic.
- تعریف variables.
- data-informed simulation.

Pooja (Execution - Business / Discovery):

- outreach.
- interviews.
- use case extraction.

## چطور task بدهی (قالب outcome)

بد:

- "برو با 10 نفر حرف بزن"

خوب:

- "تا 5 روز آینده 3 مصاحبه بگیر و خروجی را در این قالب بده: event, decision, pain point"

## ریتم کاری پیشنهادی (Weekly Loop)

1. Amir: define focus (مثلا Red Sea case)
2. James: define variables + logic
3. Pooja: validate with real people
4. Amir: جمع‌بندی و تصمیم

## نسبت حضور Amir

پیشنهاد:

- 20-30 درصد در execution.
- 70-80 درصد در direction.

## مثال هفته اول

- Amir: define scenario (Red Sea disruption)
- Pooja: 3 call, 3 use case
- James: simple delay/cost logic

## خطرها

اگر Amir:

- همه تصمیم‌ها را micro-manage کند.
- یا فقط دستور بدهد بدون outcome.

تیم کند یا disengaged می‌شود.

## جمله کلیدی

```text
Founder job = reduce uncertainty, not control everything
```
